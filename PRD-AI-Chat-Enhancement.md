# 产品需求文档 (PRD)：智能填报助手 AI 对话增强

## 1. 文档概述
**项目名称**：高考志愿智能填报系统前端 - AI 对话模块增强
**编写目的**：明确 AI 对话页面的高级交互功能与富文本展示机制，为前端开发、UI 设计以及后端接口对接提供标准依据。

## 2. 背景与目标
当前系统的 AI 对话仅支持基础的文本一问一答。为了提升高考志愿咨询的准确性与交互体验，我们需要赋予用户纠错能力（编辑与重试），降低提问门槛（推荐回复），并提供更直观的数据展现形式（图表与卡片），打造沉浸式的智能填报助手。

## 3. 核心功能详情

### 3.1 AI 对话重试 (Regenerate)
- **需求描述**：当用户对 AI 生成的回答不满意，或生成过程中发生网络中断/超时时，用户可以一键要求 AI 重新生成当前回答。
- **交互规则**：
  - 在 AI 回复气泡 (Bubble) 的操作区 (Actions) 增加“重新生成”操作入口。
  - 点击后，前端将当前的 AI 回复内容清空，并进入 Loading 状态。
  - 触发重新生成逻辑（请求后端触发重试，或前端截取上下文重新发送）。
- **异常处理**：若重试请求失败，需给出明确的全局提示 (Message/Toast)，并允许用户再次重试。

### 3.2 聊天对话编辑 (Message Edit)
- **需求描述**：允许用户修改自己发送过的历史消息，用于纠正错别字、补充条件或调整提问意图。
- **交互规则**：
  - 在用户消息气泡的操作区增加“编辑”图标。
  - 点击后，文本气泡变为可编辑的输入框，提供“取消”和“保存并发送”按钮。
  - 用户确认修改并发送后，**系统需截断（清除）该节点之后的所有历史消息（包括旧的 AI 回复及后续的对话轮次）**。
  - 将修改后的消息作为最新节点，触发 AI 流式响应。
- **依赖条件**：需要后端提供对话上下文回滚/截断能力，确保模型记忆与 UI 展示同步更新，避免上下文错乱。

### 3.3 底部智能推荐回复 (Suggested Replies)
- **需求描述**：在底部输入框上方动态展示推荐的追问或相关问题，引导用户深入咨询。
- **交互规则**：
  - 每次 AI 流式输出完全结束 (`event.type === 'done'`) 后，平滑展示 2-4 个推荐问题胶囊 (Pills)。
  - 用户点击某个胶囊后，直接将其内容作为下一条用户消息发送。
  - 用户主动在输入框中输入内容时，推荐胶囊可选择性隐藏或淡出。
- **数据流转**：
  - 方案 A：由大模型在回复末尾按照特定格式输出，前端解析后渲染。
  - 方案 B：AI 回复结束后，前端异步请求独立的 `/suggestions` 接口获取。

### 3.4 富文本与混合内容展示 (Mixed Content)
- **需求描述**：突破纯文本限制，在 AI 的回答中直接渲染交互式图表和卡片，直观呈现分数、院校等结构化数据。
- **功能拆解**：
  - **ECharts 图表嵌入**：当 AI 输出历年分数线、录取概率趋势等数据时，前端解析特定的 Markdown 代码块（如 ````echarts \n { "title": "...", "series": [...] } \n ````），并安全地渲染为 ECharts 实例。
  - **网页卡片嵌入**：当 AI 推荐特定大学或专业时，解析特定的代码块（如 ````card \n { "title": "清华大学", "cover": "..." } \n ````），渲染为图文并茂的卡片。
  - **加载态 (Loading) 容错**：在流式输出未完成、JSON 结构尚未完全闭合时，前端需捕获解析异常，并展示友好的加载动画或骨架屏，待数据接收完毕后平滑过渡为图表/卡片。

### 3.5 报告级 Markdown 样式解析
- **需求描述**：规范 AI 输出长篇填报分析报告时的排版样式，提升阅读体验。
- **功能拆解**：
  - 支持 GFM (GitHub Flavored Markdown) 标准，完全兼容表格、删除线、任务列表。
  - **样式深度定制**：标题、段落、列表的间距和排版需严格遵循项目现有的 Ant Design 设计规范（Typography 系统），并自适应明/暗色主题。
  - **表格交互优化**：针对院校对比等长表格，支持横向滚动；统一表头背景色与边框样式。

## 4. 接口需求评估 (API Requirements)
为实现上述需求，前端需与后端协同，建议后端提供或升级以下接口：
1. **对话状态截断/回滚接口**：`POST /api/v1/conversations/{id}/rollback`
   - 入参：`message_id`
   - 作用：丢弃该 ID 之后的所有对话记录，用于“编辑”历史消息。
2. **对话重试接口**：`POST /api/v1/conversations/{id}/regenerate` (可选)
   - 作用：后端直接丢弃最后一条 AI 消息并基于已有上下文重新触发流式生成。
3. **推荐问题接口**：`GET /api/v1/conversations/{id}/suggestions` (可选)
   - 作用：如果大模型无法稳定在对话末尾输出推荐问题，需要通过该接口独立获取。

## 5. 阶段实施规划 (Phases)
- **Phase 1: 基础设施构建** 
  - 引入 `react-markdown` 及相关插件，重构 `Bubble` 的渲染引擎，统一排版样式与表格展示。
- **Phase 2: 混合内容支持**
  - 约定特殊标签协议，开发 ECharts 渲染器和定制 Card 组件，与后端联调流式输出的稳定性。
- **Phase 3: 高级交互闭环**
  - 实现 UI 层的消息重试、编辑表单；对接后端的上下文回滚接口；上线底部推荐问题模块。
